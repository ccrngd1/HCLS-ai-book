"Given this patient\'s current state, what are the next valid steps?" becomes a recursive SQL query with multiple joins against condition tables. Graph databases make this query natural: start at the patient\'s current node, traverse outgoing edges where conditions are satisfied, return the reachable next nodes.\n38:\n39:### Knowledge Graph Fundamentals for Pathways\n40:\n41:A knowledge graph for clinical pathways needs several entity types:\n42:\n43:**Pathway nodes** represent clinical states or actions. Each node has a type: assessment, order, decision point, milestone, or discharge criterion. Nodes carry metadata: responsible role (physician, nurse, pharmacist), expected duration, required documentation.\n44:\n45:**Edges** represent transitions. Each edge has conditions that must be satisfied for the transition to be valid. Conditions reference patient data: lab values, vital signs, elapsed time, documented assessments, allergy status. Edges also have a type: sequential (must happen in order), parallel (can happen simultaneously), conditional (only one branch taken), or time-gated (available after a delay).\n46:\n47:**Condition nodes** represent the logic attached to edges. A condition might be simple ("CURB-65 >= 3") or compound ("temperature < 38.0 AND oral intake adequate AND oxygen saturation > 92% on room air"). Modeling conditions as first-class graph entities (rather than edge properties) lets you reuse them across pathways and reason about them independently.\n48:\n49:**Evidence nodes** link pathway decisions to their clinical evidence base. Why is the antibiotic window 4 hours? Because studies showed mortality benefit. Attaching evidence provenance to pathway nodes supports clinical governance and makes pathway updates traceable to their justification.\n50:\n51:### The Traversal Problem\n52:\n53:The core computational operation is: given a patient\'s current clinical state, which pathway nodes are active, which transitions are available, and which next actions are recommended?\n54:\n55:This requires:\n56:\n57:1. **State mapping:** Determine which pathway node(s) the patient currently occupies. A patient can be at multiple nodes simultaneously (parallel branches). This requires matching the patient\'s documented actions and results against node completion criteria.\n58:\n59:2. **Condition evaluation:** For each outgoing edge from active nodes, evaluate whether the transition conditions are met given current patient data. This means pulling real-time data from the EHR: latest labs, vitals, documented assessments, active orders.\n60:\n61:3. **Traversal:** Follow satisfied edges to identify recommended next actions. Handle parallel paths (multiple next steps available simultaneously) and exclusive branches (only one path should be taken).\n62:\n63:4. **Variance detection:** Identify when a patient\'s actual care deviates from the pathway. An order placed that isn\'t on the pathway. A pathway step that should have happened by now but hasn\'t. A branch taken that doesn\'t match the patient\'s condition data.\n64:\n65:### Temporal Reasoning\n66:\n67:Clinical pathways are deeply temporal. "Start antibiotics within 4 hours of presentation." "Reassess at 48 hours." "If no improvement after 72 hours, consider CT pulmonary angiography." The graph needs a time dimension.\n68:\n69:This means edges can have temporal constraints: minimum elapsed time before a transition is valid, maximum elapsed time before a transition becomes overdue (triggering an alert), and absolute time windows (e.g., "blood cultures before first antibiotic dose" is a sequencing constraint, not a clock constraint).\n70:\n71:Temporal reasoning in graphs is harder than it sounds. You need to track when each node was entered, compute elapsed time relative to pathway entry or node entry, and handle clock resets (patient transferred to ICU resets certain timers). Most graph databases don\'t have native temporal operators, so you\'ll implement this in your traversal logic layer.\n72:\n73:### Ontology Integration\n74:\n75:Pathways don\'t exist in isolation. The pathway graph can link to a drug ontology for allergy cross-referencing, a diagnosis hierarchy for pathway applicability rules, and a lab ontology for result interpretation. Neptune handles graph queries beautifully. Lambda scales fine. DynamoDB is fast. The hard part is getting clinical pathways out of people\'s heads and into a structured graph format.\n552:\n553:Most clinical pathways exist as Word documents or PDFs written by committee. They contain ambiguous language ("consider escalation if not improving"), implicit knowledge ("experienced clinicians know to check lactate here even though it\'s not written down"), and institutional variation ("we do it this way because Dr. Martinez prefers it"). Converting that into a formal graph with explicit conditions requires clinical informaticists who understand both the medicine and the data model. Budget more time for pathway modeling than for engineering.\n554:\n555:The versioning problem is real. When the pneumonia pathway gets updated (new antibiotic recommendations from IDSA), patients currently on version 2 need to complete under version 2. New admissions get version 3. Your system needs to handle multiple active versions simultaneously. This isn\'t hard technically (version is a property on every node and edge, and every traversal query filters by the patient\'s enrolled version), but it\'s operationally complex: who decides when to sunset old versions? What if a patient is on a pathway for 30 days and it gets updated twice? A migration function can optionally re-enroll patients on the new version if the clinical committee approves mid-pathway transitions, but that\'s a policy decision, not a technical one.\n556:\n557:Variance detection sounds great in theory. In practice, you\'ll discover that 40-60% of patients deviate from pathways for clinically appropriate reasons. The pathway says "start antibiotics within 4 hours" but the patient refused, or had an anaphylaxis history that required allergy testing first, or was in radiology for an urgent CT. Your variance reports will be noisy until you build a "justified variance" mechanism where clinicians can document why they deviated. Without it, the compliance dashboard becomes meaningless noise that everyone ignores.\n558:\n559:The condition evaluation layer is where performance problems hide. If evaluating a transition condition requires calling an EHR API to get the latest lab result, and that API takes 800ms, your "real-time CDS" is suddenly not real-time. Cache aggressively. Pre-fetch patient data when you know a CDS query is likely (patient chart opened). Accept that some conditions will be evaluated against slightly stale data and design your alerts accordingly.\n560:\n561:---\n562:\n563:## Variations and Extensions\n564:\n565:**Pathway-aware order entry.** Instead of passive CDS alerts, integrate directly with the order entry workflow. When a clinician opens the order screen for a patient on a pathway, pre-populate with the pathway\'s recommended next orders. Highlight orders that would constitute a pathway deviation. This moves from "alerting about the pathway" to "making the pathway the path of least resistance."\n566:\n567:**Outcome-linked pathway optimization.** Connect pathway variance data to patient outcomes (length of stay, readmission, mortality). 1:# Recipe 13.5: Clinical Pathway / Protocol Modeling
2:
3:**Complexity:** Medium · **Phase:** Production · **Estimated Cost:** ~$0.10 to $0.50 per pathway query (graph traversal + reasoning)
4:
5:---
6:
7:## The Problem
8:
9:A hospitalist admits a patient with community-acquired pneumonia. The hospital has a clinical pathway for this: a sequence of assessments, lab orders, antibiotic choices, escalation criteria, and discharge readiness checks. It lives in a 14-page PDF on the intranet. Maybe there's a laminated card somewhere in the nursing station. The pathway was updated six months ago, but the laminated card still shows the old antibiotic recommendations.
10:
11:This is the state of clinical pathway management at most health systems. Pathways exist as static documents. They encode complex decision logic ("if CURB-65 score >= 3, consider ICU admission; if penicillin allergy documented, substitute with respiratory fluoroquinolone") but that logic is trapped in prose. It can't be queried. It can't be traversed programmatically. It can't tell you whether a specific patient is on-pathway or has deviated. It can't alert you when a new step becomes relevant based on a lab result that just came back.
12:
13:The scale of this problem is significant. A typical academic medical center maintains 200-400 active clinical pathways covering everything from sepsis management to elective joint replacement recovery. Each pathway has decision branches, time-dependent steps, conditional orders, and escalation criteria. Keeping them current requires clinical committee review. Keeping clinicians aware of them requires training. Measuring adherence requires manual chart review.
14:
15:Order sets help, but they're flat. They give you a menu of things to order at a point in time. They don't model the temporal flow: "do this first, then wait for this result, then decide between these two branches." Clinical pathways are inherently graph-shaped: nodes are clinical states or actions, edges are transitions triggered by conditions. The moment you recognize that, the solution becomes obvious. Model them as graphs. Traverse them computationally. Use the graph to drive decision support, compliance tracking, and variance analysis.
16:
17:Let's build it.
18:
19:---
20:
21:## The Technology: Graphs for Clinical Logic
22:
23:### Why Graphs Fit Clinical Pathways
24:
25:A clinical pathway is a directed graph. Not metaphorically. Literally. Consider a simplified pneumonia pathway:
26:
27:1. Patient presents with suspected pneumonia
28:2. Order chest X-ray and blood cultures
29:3. Calculate severity score (CURB-65 or PSI)
30:4. **Decision point:** If mild (CURB-65 0-1), outpatient treatment. If moderate (2), admit to ward. If severe (3+), consider ICU.
31:5. For ward admission: start empiric antibiotics within 4 hours
32:6. Reassess at 48 hours: if improving, step down to oral. If not improving, escalate.
33:7. Discharge criteria: afebrile 24h, tolerating oral meds, oxygen saturation stable
34:
35:Each numbered item is a node. The transitions between them have conditions attached. Some transitions are time-gated ("reassess at 48 hours"). Some are event-driven ("lab result returns"). Some are conditional on patient state ("if penicillin allergy"). This is a directed acyclic graph (mostly; some pathways have loops for reassessment cycles) with typed edges.
36:
37:Relational databases can store this, but querying it is painful. "Given this patient's current state, what are the next valid steps?" becomes a recursive SQL query with multiple joins against condition tables. Graph databases make this query natural: start at the patient's current node, traverse outgoing edges where conditions are satisfied, return the reachable next nodes.
38:
39:### Knowledge Graph Fundamentals for Pathways
40:
41:A knowledge graph for clinical pathways needs several entity types:
42:
43:**Pathway nodes** represent clinical states or actions. Each node has a type: assessment, order, decision point, milestone, or discharge criterion. Nodes carry metadata: responsible role (physician, nurse, pharmacist), expected duration, required documentation.
44:
45:**Edges** represent transitions. Each edge has conditions that must be satisfied for the transition to be valid. Conditions reference patient data: lab values, vital signs, elapsed time, documented assessments, allergy status. Edges also have a type: sequential (must happen in order), parallel (can happen simultaneously), conditional (only one branch taken), or time-gated (available after a delay).
46:
47:**Condition nodes** represent the logic attached to edges. A condition might be simple ("CURB-65 >= 3") or compound ("temperature < 38.0 AND oral intake adequate AND oxygen saturation > 92% on room air"). Modeling conditions as first-class graph entities (rather than edge properties) lets you reuse them across pathways and reason about them independently.
48:
49:**Evidence nodes** link pathway decisions to their clinical evidence base. Why is the antibiotic window 4 hours? Because studies showed mortality benefit. Attaching evidence provenance to pathway nodes supports clinical governance and makes pathway updates traceable to their justification.
50:
51:### The Traversal Problem
52:
53:The core computational operation is: given a patient's current clinical state, which pathway nodes are active, which transitions are available, and which next actions are recommended?
54:
55:This requires:
56:
57:1. **State mapping:** Determine which pathway node(s) the patient currently occupies. A patient can be at multiple nodes simultaneously (parallel branches). This requires matching the patient's documented actions and results against node completion criteria.
58:
59:2. **Condition evaluation:** For each outgoing edge from active nodes, evaluate whether the transition conditions are met given current patient data. This means pulling real-time data from the EHR: latest labs, vitals, documented assessments, active orders.
60:
61:3. **Traversal:** Follow satisfied edges to identify recommended next actions. Handle parallel paths (multiple next steps available simultaneously) and exclusive branches (only one path should be taken).
62:
63:4. **Variance detection:** Identify when a patient's actual care deviates from the pathway. An order placed that isn't on the pathway. A pathway step that should have happened by now but hasn't. A branch taken that doesn't match the patient's condition data.
64:
65:### Temporal Reasoning
66:
67:Clinical pathways are deeply temporal. "Start antibiotics within 4 hours of presentation." "Reassess at 48 hours." "If no improvement after 72 hours, consider CT pulmonary angiography." The graph needs a time dimension.
68:
69:This means edges can have temporal constraints: minimum elapsed time before a transition is valid, maximum elapsed time before a transition becomes overdue (triggering an alert), and absolute time windows (e.g., "blood cultures before first antibiotic dose" is a sequencing constraint, not a clock constraint).
70:
71:Temporal reasoning in graphs is harder than it sounds. You need to track when each node was entered, compute elapsed time relative to pathway entry or node entry, and handle clock resets (patient transferred to ICU resets certain timers). Most graph databases don't have native temporal operators, so you'll implement this in your traversal logic layer.
72:
73:### Ontology Integration
74:
75:Pathways don't exist in isolation. They reference clinical concepts: diagnoses (ICD-10), procedures (CPT), medications (RxNorm), lab tests (LOINC). Your pathway graph needs to connect to these standard ontologies so that conditions like "if creatinine > 2.0" can be evaluated against actual LOINC-coded lab results from the EHR.
76:
77:This is where knowledge graphs shine over simpler representations. The pathway graph can link to a drug ontology for allergy cross-referencing, a diagnosis hierarchy for pathway applicability rules, and a lab ontology for result interpretation. These connections enable reasoning that would be impossible with a flat pathway document: "this patient is on the pneumonia pathway, but they also have CKD stage 4, which means the standard antibiotic dosing needs renal adjustment."
78:
79:### The General Architecture Pattern
80:
81:```
82:[Pathway Authoring] → [Graph Store] ← [Patient State Engine]
83:                                     ↓
84:                          [Traversal / Reasoning Engine]
85:                                     ↓
86:                    [CDS Alerts] + [Compliance Dashboard] + [Variance Reports]
87:```
88:
89:**Pathway Authoring:** Clinical informaticists model pathways as graphs using a structured editor. The editor enforces graph validity: no orphan nodes, all decision points have at least two outgoing edges, all terminal nodes are marked as endpoints.
90:
91:**Graph Store:** A graph database holds the pathway definitions. Separate from patient data. The pathway graph is the "knowledge" layer; patient state is the "data" layer.
92:
93:**Patient State Engine:** Continuously maps each patient's current clinical state to their position(s) on applicable pathways. Consumes EHR events (orders placed, results received, assessments documented) and updates the patient's pathway position.
94:
95:**Traversal / Reasoning Engine:** Given a patient's current position and clinical data, traverses the graph to determine next recommended actions, overdue steps, and available transitions. This is the query engine that powers all downstream use cases.
96:
97:**Downstream consumers:** Clinical decision support (alerts at point of care), compliance dashboards (what percentage of patients are on-pathway), and variance reports (which deviations are most common, and do they correlate with outcomes).

101:## The AWS Implementation
102:
103:### Why These Services
104:
105:**Amazon Neptune for the pathway graph store.** Neptune is AWS's graph database. It speaks two query languages: property graph (Gremlin or openCypher) and RDF (SPARQL). For clinical pathways, property graph wins easily: nodes have typed properties, edges have conditions, and traversal queries read like you'd describe the pathway out loud. Neptune handles graph storage, indexing, and query execution. It's HIPAA eligible, supports encryption at rest and in transit, and scales read replicas for query-heavy workloads. One thing to know: Neptune connections from Lambda add 200-500ms on cold start (WebSocket setup for Gremlin, or HTTP connection for openCypher). For the Traversal Engine Lambda that powers real-time CDS, configure Provisioned Concurrency of at least 2-5 instances to keep connections warm. Alternatively, use Neptune's openCypher HTTP endpoint for simpler traversal queries to avoid WebSocket overhead entirely.
106:
107:**AWS Lambda for the traversal and reasoning engine.** Pathway traversal queries are short-lived, stateless computations: receive a patient context, query Neptune for the patient's current pathway position, evaluate transition conditions against current clinical data, return recommendations. Lambda's execution model fits perfectly. For real-time CDS integration, you need sub-second response times; Neptune queries on well-indexed pathway graphs typically return in 50-200ms (warm connections), leaving plenty of Lambda execution budget.
108:
109:**Amazon DynamoDB for patient pathway state.** Each patient's current position on each applicable pathway needs fast read/write access. DynamoDB's key-value model works well here: partition key is patient ID, sort key is pathway ID, and the item contains current node(s), entry timestamps, and completed steps. This state changes with every clinical event, so write performance matters. A Global Secondary Index on `status` enables efficient queries for overdue checking at scale (avoiding full table scans).
110:
111:**Amazon EventBridge for clinical event routing.** When a lab result posts, an order is placed, or an assessment is documented, those events need to trigger pathway state evaluation. EventBridge provides the event bus that routes EHR integration events to the appropriate Lambda functions for state updates. Pair this with an SQS Dead Letter Queue on the Lambda async invocation: if the State Updater Lambda fails after retries, the clinical event lands in the DLQ rather than disappearing silently. A CloudWatch alarm on DLQ depth tells you immediately when events are being dropped.
112:
113:**Amazon S3 for pathway version storage.** Pathway definitions evolve. When a clinical committee updates the pneumonia pathway, the old version doesn't disappear; patients currently on it need to complete under their enrolled version. S3 stores versioned pathway definitions (as serialized graph structures) with lifecycle policies for retention.
114:
115:**Amazon QuickSight for compliance dashboards.** Pathway adherence metrics, variance patterns, and outcome correlations need visualization for clinical leadership. QuickSight connects to the aggregated compliance data for reporting.

176:
177:#### Walkthrough
178:
179:**Step 1: Model the pathway as a graph.** Before any code runs, you need to represent a clinical pathway as a graph structure that Neptune can store and traverse. Each pathway becomes a collection of nodes (clinical steps) connected by edges (transitions). The key design decision is how much clinical logic lives in the graph versus in the traversal code. Put conditions on edges as structured properties; put step metadata on nodes. This separation means the graph is queryable without external logic for simple traversals, but complex condition evaluation (pulling live patient data) happens in the Lambda layer. Skip this step or model it poorly, and every downstream query becomes a nightmare of application-side filtering.
180:
181:```
182:// Define the pathway graph schema.
183:// Each node represents a clinical step or decision point.
184:// Each edge represents a valid transition with conditions.
185:
186:STRUCTURE PathwayNode:
187:    id:               unique identifier (e.g., "pneumonia-v3-step-004")
188:    pathway_id:       which pathway this belongs to
189:    pathway_version:  version number (pathways evolve over time)
190:    node_type:        one of [assessment, order, decision_point, milestone, discharge_criterion]
191:    name:             human-readable step name (e.g., "Calculate CURB-65 Score")
192:    description:      clinical description of what this step involves
193:    responsible_role: who performs this (physician, nurse, pharmacist, respiratory_therapy)
194:    expected_duration_hours: how long this step typically takes (null if instantaneous)
195:    required_documentation: what must be documented to mark this step complete
196:    parallel_group:   identifier for steps that can execute simultaneously (null if sequential)
197:
198:STRUCTURE PathwayEdge:
199:    from_node:        source node ID
200:    to_node:          target node ID
201:    edge_type:        one of [sequential, conditional, time_gated, parallel_start, parallel_join]
202:    conditions:       list of Condition objects that must ALL be true for this transition
203:    priority:         for conditional edges from same node, evaluation order (lower = first)
204:    max_time_hours:   if set, this transition becomes "overdue" after this many hours
205:
206:STRUCTURE Condition:
207:    condition_type:   one of [lab_value, vital_sign, elapsed_time, assessment_complete,
208:                              order_placed, allergy_check, diagnosis_present]
209:    parameter:        what to check (e.g., LOINC code for lab, vital sign name)
210:    operator:         one of [gt, gte, lt, lte, eq, neq, exists, not_exists]
211:    value:            threshold or expected value
212:    time_window_hours: how recent the data must be (null = any time)

215:**Step 2: Load pathway into Neptune.** Once you have the pathway modeled as structured data, load it into Neptune as a property graph. Each PathwayNode becomes a vertex with properties. Each PathwayEdge becomes an edge with properties. Conditions are stored as JSON properties on edges (for simple conditions) or as separate vertices linked to edges (for complex, reusable conditions). The loading process should validate graph integrity: every edge references existing nodes, every decision point has at least two outgoing conditional edges, and every pathway has exactly one start node and at least one terminal node.
216:
217:```
218:FUNCTION load_pathway_to_neptune(pathway_definition):
219:    // Validate the pathway graph structure before loading.
220:    // Catch structural errors here rather than discovering them at query time.
221:    VALIDATE:
222:        - exactly one node with node_type = "start"
223:        - at least one node with node_type = "discharge_criterion" or "terminal"
224:        - every decision_point node has >= 2 outgoing conditional edges
225:        - no orphan nodes (every node reachable from start)
226:        - no dangling edges (both endpoints exist)
227:
228:    // Create vertices for each pathway node.
229:    FOR each node in pathway_definition.nodes:
230:        ADD VERTEX to Neptune with:
231:            label     = node.node_type
232:            id        = node.id
233:            properties = {
234:                pathway_id:       node.pathway_id,
235:                pathway_version:  node.pathway_version,
236:                name:             node.name,
237:                description:      node.description,
238:                responsible_role: node.responsible_role,
239:                expected_duration_hours: node.expected_duration_hours,
240:                parallel_group:   node.parallel_group
241:            }
242:
243:    // Create edges for each transition.
244:    FOR each edge in pathway_definition.edges:
245:        ADD EDGE to Neptune from edge.from_node to edge.to_node with:
246:            label      = edge.edge_type
247:            properties = {
248:                conditions:     serialize(edge.conditions) as JSON,
249:                priority:       edge.priority,
250:                max_time_hours: edge.max_time_hours
251:            }
252:
253:    // Store the full pathway definition in S3 for versioning and audit.
254:    WRITE pathway_definition to S3 at:
255:        bucket: "clinical-pathways"
256:        key:    "{pathway_id}/v{pathway_version}/definition.json"

259:**Step 3: Map patient to pathway position.** When a patient is enrolled on a pathway (either manually by a clinician or automatically based on admission diagnosis), the system needs to track their current position. This means determining which node(s) they currently occupy. A patient can be at multiple nodes simultaneously if the pathway has parallel branches. The state record in DynamoDB captures: which nodes are active, when each was entered, and which nodes have been completed.
260:
261:```
262:FUNCTION initialize_patient_on_pathway(patient_id, pathway_id, pathway_version):
263:    // Find the start node for this pathway version.
264:    start_node = QUERY Neptune:
265:        "Find vertex where pathway_id = {pathway_id}
266:         AND pathway_version = {pathway_version}
267:         AND label = 'start'"
268:
269:    // Create the patient's pathway state record in DynamoDB.
270:    // This record will be updated every time a clinical event advances the patient.
271:    WRITE to DynamoDB table "patient-pathway-state":
272:        partition_key = patient_id
273:        sort_key      = pathway_id
274:        attributes    = {
275:            pathway_version:   pathway_version,
276:            enrolled_at:       current UTC timestamp,
277:            active_nodes:      [start_node.id],          // currently at the start
278:            node_entry_times:  {start_node.id: current UTC timestamp},
279:            completed_nodes:   [],                        // nothing completed yet
280:            completed_edges:   [],                        // no transitions taken yet
281:            status:            "active"                   // active, completed, or withdrawn
282:        }
283:
284:FUNCTION advance_patient_state(patient_id, pathway_id, completed_node_id, next_node_id):
285:    // Called when a transition is confirmed (conditions met, action taken).
286:    // Atomically updates the patient's position on the pathway.
287:    UPDATE DynamoDB record for (patient_id, pathway_id):
288:        REMOVE completed_node_id from active_nodes
289:        ADD next_node_id to active_nodes
290:        ADD completed_node_id to completed_nodes
291:        SET node_entry_times[next_node_id] = current UTC timestamp
292:        ADD {from: completed_node_id, to: next_node_id, at: timestamp} to completed_edges
293:```

295:**Step 4: Evaluate transitions on clinical events.** This is the core reasoning step. When a clinical event occurs (lab result posted, order completed, assessment documented), the system checks whether any transitions from the patient's current active nodes are now satisfiable. This requires pulling the patient's current clinical data and evaluating each outgoing edge's conditions. If conditions are met, the transition fires and the patient advances. Note: multiple pathway versions coexist in Neptune simultaneously. Every traversal query must include the patient's enrolled version as a filter to avoid returning nodes from the wrong version.
296:
297:```
298:FUNCTION on_clinical_event(event):
299:    // A clinical event arrived: lab result, order status change, vital sign, etc.
300:    patient_id = event.patient_id
301:
302:    // Get all active pathway enrollments for this patient.
303:    pathway_states = QUERY DynamoDB:
304:        "All records where partition_key = {patient_id} AND status = 'active'"
305:
306:    FOR each state in pathway_states:
307:        FOR each active_node_id in state.active_nodes:
308:
309:            // Get outgoing edges from this node, ordered by priority.
310:            // MUST filter by enrolled pathway version to avoid cross-version traversal.
311:            outgoing_edges = QUERY Neptune:
312:                "Find all edges FROM {active_node_id}
313:                 WHERE pathway_id = {state.pathway_id}
314:                 AND pathway_version = {state.pathway_version}
315:                 ORDER BY priority ASC"
316:
317:            FOR each edge in outgoing_edges:
318:                // Evaluate all conditions on this edge against current patient data.
319:                conditions_met = evaluate_conditions(
320:                    edge.conditions,
321:                    patient_id,
322:                    state.node_entry_times[active_node_id]  // for elapsed time calculations
323:                )
324:
325:                IF conditions_met:
326:                    // Transition is valid. Advance the patient.
327:                    advance_patient_state(
328:                        patient_id,
329:                        state.pathway_id,
330:                        active_node_id,
331:                        edge.to_node
332:                    )
333:
334:                    // For exclusive decision points, stop evaluating other edges.
335:                    // The first satisfied condition wins (priority ordering matters).
336:                    IF edge.edge_type == "conditional":
337:                        BREAK
338:
339:FUNCTION evaluate_conditions(conditions, patient_id, node_entry_time):
340:    // Check every condition in the list. ALL must be true (AND logic).
341:    FOR each condition in conditions:
342:        IF condition.condition_type == "lab_value":
343:            // Pull the most recent lab result for this LOINC code.
344:            lab = get_latest_lab(patient_id, condition.parameter, condition.time_window_hours)
345:            IF lab is null OR NOT compare(lab.value, condition.operator, condition.value):
346:                RETURN false
347:
348:        ELSE IF condition.condition_type == "elapsed_time":
349:            // Check if enough time has passed since entering the current node.
350:            elapsed = hours_since(node_entry_time)
351:            IF NOT compare(elapsed, condition.operator, condition.value):
352:                RETURN false
353:
354:        ELSE IF condition.condition_type == "assessment_complete":
355:            // Check if a specific assessment has been documented.
356:            assessment = get_assessment(patient_id, condition.parameter)
357:            IF assessment is null:
358:                RETURN false
359:
360:        ELSE IF condition.condition_type == "allergy_check":
361:            // Check patient allergy list against a drug class.
362:            has_allergy = check_allergy(patient_id, condition.parameter)
363:            IF has_allergy != (condition.operator == "exists"):
364:                RETURN false
365:
366:        // ... additional condition types as needed
367:
368:    RETURN true  // all conditions satisfied
369:```

371:**Step 5: Detect overdue transitions and variances.** Not all pathway deviations are triggered by events. Some are the absence of events: a step that should have happened by now but hasn't. A scheduled Lambda runs periodically (every 15-30 minutes) to check for overdue transitions and off-pathway actions. This is where compliance monitoring lives. Use a DynamoDB GSI on `status` (with `oldest_node_entry_time` as sort key) to query only active states efficiently, rather than scanning the entire table.
372:
373:```
374:FUNCTION check_overdue_transitions():
375:    // Query all active patient pathway states using the status GSI.
376:    // This avoids a full table scan, which won't scale beyond ~500 patients.
377:    // Paginate through all results (DynamoDB returns max 1MB per call).
378:
379:    active_states = QUERY DynamoDB GSI "status-index"
380:        WHERE status = "active"
381:        ORDER BY oldest_node_entry_time ASC
382:        // Paginate: loop on LastEvaluatedKey until all pages processed.
383:
384:    FOR each state in active_states:
385:        FOR each active_node_id in state.active_nodes:
386:            node_entry_time = state.node_entry_times[active_node_id]
387:
388:            // Check outgoing edges for time-gated transitions that are overdue.
389:            // Filter by enrolled pathway version.
390:            outgoing_edges = QUERY Neptune:
391:                "Find edges FROM {active_node_id}
392:                 WHERE pathway_version = {state.pathway_version}
393:                 AND max_time_hours IS NOT NULL"
394:
395:            FOR each edge in outgoing_edges:
396:                elapsed = hours_since(node_entry_time)
397:                IF elapsed > edge.max_time_hours:
398:                    // This transition is overdue. The patient should have moved past this step.
399:                    generate_alert(
400:                        type:        "overdue_pathway_step",
401:                        patient_id:  state.patient_id,
402:                        pathway_id:  state.pathway_id,
403:                        node_id:     active_node_id,
404:                        node_name:   get_node_name(active_node_id),
405:                        hours_overdue: elapsed - edge.max_time_hours,
406:                        expected_action: get_node_name(edge.to_node)
407:                    )
408:
409:FUNCTION detect_off_pathway_action(patient_id, action_taken):
410:    // Called when an order or action is placed. Check if it's on the pathway.
411:    active_states = QUERY DynamoDB for patient_id WHERE status = "active"
412:
413:    FOR each state in active_states:
414:        // Get all nodes in this pathway that represent the action taken.
415:        matching_nodes = QUERY Neptune:
416:            "Find vertices in pathway {state.pathway_id}
417:             WHERE pathway_version = {state.pathway_version}
418:             AND action_code = {action_taken.code}"
419:
420:        IF matching_nodes is empty:
421:            // Action is not part of this pathway at all. Log as variance.
422:            log_variance(
423:                type:       "off_pathway_action",
424:                patient_id: patient_id,
425:                pathway_id: state.pathway_id,
426:                action:     action_taken,
427:                timestamp:  current timestamp
428:            )

431:**Step 6: Query for CDS recommendations.** When a clinician opens a patient's chart or an order entry screen, the system queries for current pathway recommendations. This is the real-time traversal that powers point-of-care decision support. It returns: what the patient should do next, what's overdue, and what branches are available based on current data.
432:
433:```
434:FUNCTION get_pathway_recommendations(patient_id, pathway_id):
435:    // Called at point of care. Must return in < 500ms for CDS integration.
436:
437:    state = GET from DynamoDB (patient_id, pathway_id)
438:    IF state is null OR state.status != "active":
439:        RETURN empty recommendations
440:
441:    recommendations = []
442:
443:    FOR each active_node_id in state.active_nodes:
444:        node = GET vertex from Neptune by id = active_node_id
445:
446:        // Get all outgoing edges and evaluate which transitions are available.
447:        // Filter by enrolled pathway version.
448:        outgoing_edges = QUERY Neptune:
449:            "Find edges FROM {active_node_id}
450:             WHERE pathway_version = {state.pathway_version}
451:             ORDER BY priority"
452:
453:        available_transitions = []
454:        FOR each edge in outgoing_edges:
455:            conditions_met = evaluate_conditions(edge.conditions, patient_id,
456:                                                  state.node_entry_times[active_node_id])
457:            target_node = GET vertex from Neptune by id = edge.to_node
458:
459:            available_transitions.append({
460:                target_node:    target_node.name,
461:                target_type:    target_node.node_type,
462:                conditions_met: conditions_met,
463:                conditions:     describe_conditions_human_readable(edge.conditions),
464:                edge_type:      edge.edge_type
465:            })
466:
467:        // Check if current node is overdue.
468:        elapsed = hours_since(state.node_entry_times[active_node_id])
469:        is_overdue = any(edge.max_time_hours != null AND elapsed > edge.max_time_hours
470:                         for edge in outgoing_edges)
471:
472:        recommendations.append({
473:            current_step:           node.name,
474:            current_step_type:      node.node_type,
475:            responsible_role:       node.responsible_role,
476:            time_in_step_hours:     elapsed,
477:            is_overdue:             is_overdue,
478:            available_transitions:  available_transitions
479:        })
480:
481:    RETURN {
482:        patient_id:      patient_id,
483:        pathway_id:      pathway_id,

485:        status:          state.status,
486:        recommendations: recommendations,
487:        completed_steps: length(state.completed_nodes),

549:## The Honest Take
550:
551:Here's what will surprise you about this project: the technology is the easy part. Neptune handles graph queries beautifully. Lambda scales fine. DynamoDB is fast. The hard part is getting clinical pathways out of people's heads and into a structured graph format.
552:
553:Most clinical pathways exist as Word documents or PDFs written by committee. They contain ambiguous language ("consider escalation if not improving"), implicit knowledge ("experienced clinicians know to check lactate here even though it's not written down"), and institutional variation ("we do it this way because Dr. Martinez prefers it"). Converting that into a formal graph with explicit conditions requires clinical informaticists who understand both the medicine and the data model. Budget more time for pathway modeling than for engineering.
554:
555:The versioning problem is real. When the pneumonia pathway gets updated (new antibiotic recommendations from IDSA), patients currently on version 2 need to complete under version 2. New admissions get version 3. Your system needs to handle multiple active versions simultaneously. This isn't hard technically (version is a property on every node and edge, and every traversal query filters by the patient's enrolled version), but it's operationally complex: who decides when to sunset old versions? What if a patient is on a pathway for 30 days and it gets updated twice? A migration function can optionally re-enroll patients on the new version if the clinical committee approves mid-pathway transitions, but that's a policy decision, not a technical one.
556:
557:Variance detection sounds great in theory. In practice, you'll discover that 40-60% of patients deviate from pathways for clinically appropriate reasons. The pathway says "start antibiotics within 4 hours" but the patient refused, or had an anaphylaxis history that required allergy testing first, or was in radiology for an urgent CT. Your variance reports will be noisy until you build a "justified variance" mechanism where clinicians can document why they deviated. Without it, the compliance dashboard becomes meaningless noise that everyone ignores.
558:
559:The condition evaluation layer is where performance problems hide. If evaluating a transition condition requires calling an EHR API to get the latest lab result, and that API takes 800ms, your "real-time CDS" is suddenly not real-time. Cache aggressively. Pre-fetch patient data when you know a CDS query is likely (patient chart opened). Accept that some conditions will be evaluated against slightly stale data and design your alerts accordingly.

563:## Variations and Extensions
564:
565:**Pathway-aware order entry.** Instead of passive CDS alerts, integrate directly with the order entry workflow. When a clinician opens the order screen for a patient on a pathway, pre-populate with the pathway's recommended next orders. Highlight orders that would constitute a pathway deviation. This moves from "alerting about the pathway" to "making the pathway the path of least resistance."
566:
567:**Outcome-linked pathway optimization.** Connect pathway variance data to patient outcomes (length of stay, readmission, mortality). Over time, identify which variances correlate with better outcomes. Maybe the pathway says 48-hour reassessment, but patients who get reassessed at 36 hours have shorter stays. Feed this back to the clinical committee for evidence-based pathway updates. This turns the knowledge graph into a learning system.
568:
569:**Multi-pathway coordination.** Patients often qualify for multiple pathways simultaneously (pneumonia pathway AND heart failure pathway AND diabetes management pathway). Build a coordination layer that detects conflicts between pathways (one says "aggressive IV fluids," the other says "fluid restriction") and surfaces them for clinical resolution. This requires cross-pathway edge analysis in Neptune.
