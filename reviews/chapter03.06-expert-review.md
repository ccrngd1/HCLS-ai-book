## Overall Assessment

**Verdict: PASS**

This is the most ambitious architectural teaching in Chapter 3 to date and one of the densest substantive treatments in the cookbook. The Monday-morning SIU whiteboard vignette (three clinics in the same strip mall, four providers from the same training cohort, two DME suppliers with overlapping ownership, one toxicology lab receiving 80% of referrals, $11.7M paid, ~400 patients, average $29k per patient versus $3.2k peer norm) is publication-ready voice and lands the operational reality of organized scheme detection in two paragraphs. The fraud/waste/abuse terminology disambiguation (intentional deception vs overutilization without deception vs ambiguous middle-ground billing) is correct, legally grounded, and operationally accurate; framing most real cases as mixtures along the spectrum is exactly how senior SIU practitioners describe the problem.

The scheme taxonomy section (phantom billing, upcoding, unbundling, medically unnecessary services, kickbacks and self-referral, identity theft and credential abuse, collusive networks, patient-side abuse, billing-mill patterns) is the strongest single teaching anchor in the recipe, and the explicit framing that different schemes require structurally different detection approaches is the design-constraint-up-front teaching the cookbook is built on. Each scheme type names operational detection clues that match what every senior SIU analyst would describe (modifier 59 above peer norms for unbundling; service date patterns inconsistent with facility hours for phantom billing; high referral concentration to commonly-owned entities for kickbacks; uniform documentation across patients for billing mills). The legal grounding is unusually substantive: 42 USC 1320a-7b for the Anti-Kickback Statute, 42 USC 1395nn for Stark Law, 31 USC 3729-3733 for the False Claims Act, 42 CFR 422.504(h) and 42 CFR 438.608 for Medicare Advantage and Medicaid managed care fraud reporting, 42 CFR 411.354 for physician self-referral, 45 CFR 164.512(f) for HIPAA disclosures to law enforcement. Names the specific statutory anchors that distinguish actionable cases from "this looks weird" hunches.

#### Finding S1: Patient Node ID Hashed in Two Places but Bypassed in the `billed` Edge Construction

- **Severity:** MEDIUM
- **Expert:** Security / Pedagogy
- **Location:** Step 3 `refresh_graph` pseudocode, the `// Upsert service edges.` block. Specifically the `// Billed-by edge: organization -> claim` upsert.
- **Problem:** The patient-vertex upsert correctly hashes the patient ID (`Neptune.UpsertVertex(label = "Patient", id = hash_patient_id(patient_id), ...)`), and the `rendered` edge correctly hashes (`to = hash_patient_id(claim.patient_id)`). But the `billed` edge upsert says:

  ```
  // Billed-by edge: organization -> claim
  Neptune.UpsertEdge(
      label = "billed",
      from  = claim.billing_organization_canonical_id,
      to    = hash_patient_id(claim.patient_id),
      properties = {
          claim_id:        claim.claim_id,
          billed_amount:   claim.claim_total_billed,
          paid_amount:     claim.claim_total_paid
      }
  )
  ```

  The to-end is `hash_patient_id(claim.patient_id)` (correctly hashed), but the comment says "organization -> claim" while the edge actually goes "organization -> patient (hashed)." This is a teaching-fidelity bug: the comment promises a `claim` node and the code produces a patient-target. The recipe's prose elsewhere lists `claims` as a node type ("Nodes for every entity, edges for every relationship") and the high-value-query examples include "Patient-sharing between providers" which depends on patient nodes existing as graph entities. But there is no claim-node upsert anywhere in Step 3, so a graph constructed strictly from this pseudocode has no claim vertices despite the prose declaring them.

  This propagates into Step 6 `run_graph_analytics`: the `referrals = Neptune.Query(...)` query "find all entities that this provider has referred to" works because `referred` edges are upserted directly between providers. But community detection (Louvain) on a graph that mixes provider-to-patient `rendered` edges with organization-to-patient `billed` edges produces communities centered on patients, which is operationally backwards: the recipe correctly identifies patient-sharing as a graph signal, but it's a derived metric on the provider/organization graph, not the unit of community detection. A community detection algorithm run on the constructed graph will discover dense patient-centered subgraphs (a single patient with many provider-rendered edges and many organization-billed edges) rather than the provider-and-organization collusive networks the recipe's prose describes.

  More concretely: the recipe's Step 3 doesn't actually construct claims as graph entities and doesn't construct edges between claims and the entities that bill or rendered for them. The claim_id appears only as an edge property. A Gremlin traversal "find all claims for which provider X is the rendering provider" cannot run against this graph because there are no claim vertices. A reader who treats the pseudocode as the implementation specification will not know to add claim vertices, and the resulting graph will not support the queries the recipe's prose describes.

  This is a teaching-fidelity gap with consequences for the operational pattern: a graph that mixes rendered-by patient-target edges with billed-by patient-target edges and lacks claim vertices will not support the high-value queries the recipe identifies (claim-volume by community, payment flows through organizations, claim-level temporal coordination patterns).

- **Fix:** Two coordinated edits:
  1. Update Step 3 `refresh_graph` to upsert claim vertices and to construct edges between providers, organizations, claims, and patients as separate edge types:

     ```
     // Upsert claim vertices.
     FOR each claim in new_claims:
         Neptune.UpsertVertex(
             label = "Claim",
             id    = claim.claim_id,
             properties = {
                 service_date:    claim.primary_service_date,
                 billed_amount:   claim.claim_total_billed,
                 paid_amount:     claim.claim_total_paid,
                 primary_cpt:     claim.primary_cpt,
                 primary_diagnosis: claim.primary_diagnosis
             }
         )

     // Service edges.
     FOR each claim in new_claims:
         // Provider-to-claim: which provider rendered the service on this claim.
         Neptune.UpsertEdge(
             label = "rendered_on_claim",
             from  = claim.rendering_provider_canonical_id,
             to    = claim.claim_id,
             properties = { service_date: claim.primary_service_date }
         )
         // Organization-to-claim: which organization billed for this claim.
         Neptune.UpsertEdge(
             label = "billed_for_claim",
             from  = claim.billing_organization_canonical_id,
             to    = claim.claim_id,
             properties = { billed_amount: claim.claim_total_billed, paid_amount: claim.claim_total_paid }
         )
         // Claim-to-patient: which patient was the subject of the claim.
         Neptune.UpsertEdge(
             label = "for_patient",
             from  = claim.claim_id,
             to    = hash_patient_id(claim.patient_id),
             properties = {}
         )
         // Referred-by edge stays provider-to-provider as in the original.
         IF claim.referring_npi is not null:
             Neptune.UpsertEdge(
                 label = "referred",
                 from  = claim.referring_provider_canonical_id,
                 to    = claim.rendering_provider_canonical_id,
                 properties = { ... }
             )
     ```
  2. Update Step 6 `run_graph_analytics`'s community-detection projection to be explicit about which node types and edge types participate in the projection. Communities should be discovered over a projected graph that excludes patient nodes from the community-membership calculation but includes patient-shared-by-providers as a derived edge weight or as a post-detection enrichment. Otherwise community detection finds patient-centered communities rather than provider-and-organization collusive networks. Add a one-paragraph note to the Graph-Is-The-Secret-Sauce subsection naming projection design as a first-class concern.

#### Finding S2: SNS/EventBridge `CaseReady.triage` Notification Payload PHI Minimization Not Stated

- **Severity:** MEDIUM
- **Expert:** Security
- **Location:** Step 7 `on_flag_event` (`EventBridge.PutEvent(bus = "fwa-workflow", source = "evidence-aggregator", detail_type = "CaseReady.triage", detail = { case_id: case.id, priority_score: case.priority_score })`); Step 7's `OpenSearch.Index("fwa-cases", case)` call where `case` includes the full evidence_bundle.
- **Problem:** The Step 7 EventBridge event for case-ready-for-triage carries a minimal payload (case_id + priority_score), which is the right discipline. But two observations:

  1. The OpenSearch index of the case (which carries the full evidence_bundle including representative_claims, peer_comparison, subgraph, prior_flags_on_entity, prior_case_outcomes_on_entity, estimated_dollar_impact, legal_basis_flags, documentation_attached) and the DynamoDB upsert of the case state both happen before the EventBridge event fires. A subscriber that receives the CaseReady.triage event and immediately fetches by case_id will fetch the full PHI-bearing case payload from OpenSearch or DynamoDB. The discipline of "the notification carries minimum data; the consumer fetches by ID" is correct, but the consumer fetch path is itself PHI-bearing and the IAM scope on each subscriber matters. The recipe does not break out per-subscriber IAM scope for OpenSearch and DynamoDB fetches against the case stores.

  2. The recipe's Step 7 notes that `case.flags.append(...)` includes the full `flag` (which may include `representative_claims`, `excluded_entity_id`, `service_date`, `death_date`, `dollar_impact`) and the OpenSearch index call serializes the full case. The OpenSearch index becomes a comprehensive PHI store. The recipe's prose says "OpenSearch's k-NN vector search supports embedding-based similarity on provider vectors. Fine-grained access control is critical because SIU data has tighter access restrictions than general payment integrity data," which is the right discipline, but the per-investigator-assignment access scope (the discipline the prose describes) is not in the pseudocode.

  This is the same chapter-3-settled "minimum-PHI-in-notification" pattern as Recipes 3.1, 3.3, 3.4, 3.5, with an FWA-specific twist: the flag detail (excluded_entity_id, post-mortem-billing patient_id with death_date, kickback ownership cascade) is itself sensitive in ways that go beyond per-patient PHI. A flag that says "Provider X has been excluded from federal programs since date Y but continues to bill" is sensitive provider information; a flag that says "Patient Z's services post-date their date of death" includes the patient's date of death; an ownership-cascade flag exposes corporate-officer information. The investigator-tier-of-access discipline matters across all three.
- **Fix:** Three coordinated edits:
  1. State explicitly in Step 7 that the EventBridge `CaseReady.triage` event carries only `case_id` and `priority_score` and that subscribers must fetch by ID through their own IAM-scoped paths. Add: "PHI does not transit through EventBridge; subscribers (case-management UI, audit indexer, notification service) fetch by case_id through their IAM-scoped read paths into OpenSearch and DynamoDB. The notification channels (email, Teams, Slack) carry only case_id and the routing tier; even the entity name and the dollar impact are not in the notification subject because they are operationally sensitive and may be visible on lock screens."
  2. Strengthen the Prerequisites IAM row with per-investigator-assignment scope: "Investigator roles read case data scoped by case ownership (case_id IN assigned_cases for the investigator's role); supervisor roles can override the assignment scope but with CloudTrail data-event audit. Analyst roles can read all cases at low-detail (case-id, priority-score, dollar-impact, scheme hypothesis) but require case-assignment to read full evidence bundles. Pathologist and clinical-reviewer roles have read-only scope tied to specific cases that have been routed for clinical review. Compliance roles read all cases for audit purposes; their reads are logged at higher granularity."
  3. State explicitly that OpenSearch fine-grained access control rules are evaluated against the investigator's role and case-assignment claims, and that case-document indexing includes a `case_assignments` field that the access-control rules query against. This is the architectural backstop for the prose discipline.

#### Finding S3: Subgroup Data Governance for Fairness Monitoring Not Specified at the Infrastructure Level

- **Severity:** MEDIUM
- **Expert:** Security / Compliance
- **Location:** "Why This Isn't Production-Ready" Fairness, bias, and equity monitoring bullet ("Detection rates by specialty, by geography, by patient demographics, by provider race/ethnicity when identifiable, by practice setting... Mitigations include case-mix adjustment, subgroup performance review, and threshold-tuning review specifically for equity").
- **Problem:** The recipe is unusually clear about subgroup dimensions: provider race/ethnicity (when identifiable), patient demographics, geography, practice setting (safety net, rural, FQHC). Naming these explicitly is the right discipline. But the architectural artifacts that make subgroup monitoring binding rather than aspirational are not specified: where the demographic and provider-attribute store lives, who has read access, how it joins to flag events and case outcomes, what the audit trail for subgroup queries looks like, what IAM scope the QuickSight dashboard role and the retraining job role need on demographic data, and how the dashboard avoids exposing row-level demographic data to dashboard viewers.

  Provider race/ethnicity data is unusually sensitive in the FWA context because the historical pattern of disproportionate enforcement against minority-owned practices is a well-documented concern. Some state laws restrict secondary use of demographic data more tightly than HIPAA restricts PHI per se. The recipe's framing-level treatment is correct; the architectural backstop that operationalizes it is missing.

  The case-mix adjustment is itself a methodological-and-data-architecture concern: it requires patient-level acuity, comorbidity, and demographic data joined to provider attribution, which is a substantial subgroup-attribute-rich data flow that needs governance. Same finding shape as Recipes 3.2 S2, 3.3 S2, 3.4 S2, 3.5 S2 (now five recipes deep).
- **Fix:** Add a "Subgroup data access" paragraph or row to Prerequisites: "Subgroup performance and detection-rate monitoring requires read access to provider attributes (race, ethnicity when identifiable, geography, practice setting) and patient demographic attributes (age band, sex, race, ethnicity, insurance type, language). These attributes may be governed differently from clinical PHI in some regulatory regimes (state laws often restrict secondary use of race/ethnicity data); restrict read access to the demographic-and-attribute stores to the retraining job role and the fairness-monitoring dashboard role, and audit subgroup queries via CloudTrail data events. The QuickSight dashboard backed by Athena should query an aggregated subgroup-metrics table (flag rate by provider specialty by geography by practice setting; case-disposition rates by patient demographic; overpayment-recovery rates by provider attribute), not the raw demographic-joined flag archive, so that dashboard-user access does not require row-level read on the subgroup attributes. Case-mix adjustment requires patient-level demographic data joined to provider attribution; this join occurs in a controlled environment with output limited to aggregated metrics." Strengthen the IAM row with the per-role scope: retraining job role gets `glue:GetTable` and `s3:GetObject` only on the demographic-joined view; fairness-monitoring dashboard role gets read-only on the aggregated subgroup-metrics table.

#### Finding S4: Legal-Privilege Architecture Acknowledged in Prose but No Infrastructure Primitive Specified

- **Severity:** MEDIUM
- **Expert:** Security / Healthcare Compliance
- **Location:** "Why This Isn't Production-Ready" Legal and privileged architecture bullet ("Depending on the organizational structure of the SIU (under legal counsel, under compliance, under operations), the detection infrastructure may need to be partitioned from general analytics environments. Legal privilege considerations affect what's stored where, who has access, how data requests are handled. Coordinate with the general counsel before architectural decisions."); Prerequisites Legal Privilege Architecture row ("If the SIU operates under legal privilege, the case management data may need to be in an account or VPC isolated from general analytics, with access controlled by legal counsel. Coordinate with the general counsel's office before designing the architecture.").
- **Problem:** The recipe correctly identifies that SIU operations under legal privilege have implications for data architecture. But the specific infrastructure primitives that operationalize the privilege boundary are not specified: separate AWS account in an Organization Unit governed by general counsel; separate VPC with no peering to general analytics environments; separate KMS keys controlled by GC-administered roles with key-policy allow lists that exclude general analytics roles; distinct CloudTrail trail with delivery to a GC-controlled S3 bucket; distinct OpenSearch domain and DynamoDB tables for case data with explicit IAM-and-data-classification labels; SCP-level controls preventing data export from the privileged-environment account. The general consequence of getting these primitives wrong is that work product may lose privilege protection during litigation discovery, which is the exact risk the legal-privilege architecture is meant to prevent.

  The recipe's framing ("coordinate with the general counsel before architectural decisions") is correct discipline but it does not give the architect the infrastructure primitives to bring to that conversation. A first-time builder who reads this paragraph will have a discussion with legal counsel and not know what infrastructure-level controls to propose.
- **Fix:** Expand the Prerequisites Legal Privilege Architecture row to name the infrastructure primitives:

  "If the SIU operates under legal privilege (general counsel oversees the SIU and investigation work product is attorney work product), the detection infrastructure must be architected to preserve the privilege. The infrastructure primitives:

  - Separate AWS account in an Organization Unit administered by general counsel; account-level SCPs prevent data export, prevent CloudTrail disabling, and prevent key-policy modification
  - Separate VPC with no peering to general analytics environments; egress strictly through documented PrivateLink endpoints to specific shared services
  - Separate customer-managed KMS keys whose key policies allow access only to GC-administered roles; key policies must exclude analytics-engineer and data-platform roles even at the broker level
  - Distinct CloudTrail trail with delivery to a GC-controlled S3 bucket in a separate account; logs are subject to FCA-statute-of-limitations retention
  - Distinct OpenSearch domain and DynamoDB tables for case data; data classification tags ('PRIVILEGED') applied at the resource level
  - SCP-level prevention of S3 cross-account access from the privileged environment; only specifically-allow-listed cross-account paths permitted
  - Coordinate with the general counsel's office before designing the architecture; the GC's access control posture (read-only audit, full administrative, dedicated investigators) determines specific role-design choices."

  Add a one-line note to the Why This Isn't Production-Ready legal-privilege bullet that points at the Prerequisites row for the infrastructure primitives.

#### Finding A1: Outcome-Event Idempotency Not Modeled at the Evidence-Aggregator Lambda

- **Severity:** MEDIUM
- **Expert:** Architecture
- **Location:** Step 7 `on_flag_event` (consumed via EventBridge from `fwa-flags` bus); Architecture Diagram (`X[Amazon EventBridge\nfwa-flags] --> Y[AWS Lambda\nevidence-aggregator]`).
- **Problem:** The architecture is event-driven from the four detector layers (rules, statistical, graph, supervised re-ranker) to the evidence-aggregator Lambda via EventBridge. EventBridge guarantees at-least-once delivery; Lambda asynchronous invocation also retries on failure. If EventBridge redelivers a flag event (the Lambda's first attempt timed out, the detector's PutEvents call retried after a transient error, or any of the failure modes documented in the recurring trigger-idempotency findings across Recipes 2.4-2.10 and 3.1-3.5), the evidence-aggregator runs twice on the same flag. Each run:

  1. **Appends the flag to `case.flags`** (`case.flags.append({ flag: flag, detected_at: NOW(), detector: flag.detector_source })`). The append produces duplicate flag entries on the case; the case's combined-dollar-impact computation (`compute_combined_dollar_impact(case.flags)`) double-counts; the priority score (`compute_priority_score(case.evidence_bundle, program_priorities())`) is derived from the doubled dollar impact and is therefore distorted; cases that should rank below the case-review threshold rise above it because of doubled flags; the investigator queue depth is artificially inflated.

  2. **Re-fetches the subgraph and peer comparison** (`fetch_subgraph(target, hops = 3, weighted_by = "paid_amount")`, `build_peer_comparison(target, flag)`). These are expensive operations against Neptune and the feature store; redelivery doubles the cost without producing additional value.

  3. **Re-publishes the `CaseReady.triage` event** if the doubled priority score crosses the review threshold. The triage queue now has duplicate triage events for the same case, the analyst tooling displays the case twice, and the analyst-workflow-tracking metrics are corrupted.

  Same recurring trigger-idempotency pattern as Recipes 2.4-2.10, 3.1, 3.2, 3.3, 3.4, 3.5 (twelfth consecutive recipe). The fix template is the same.

  For Recipe 3.6 specifically the consequence is calibrated to the regulatory-referral pathway: a doubled priority score on a case that is on the edge of the regulatory-referral threshold can flip the case from "investigation only" to "regulatory referral" or vice versa. For Medicare Advantage organizations under 42 CFR 422.504(h), the timing and accuracy of the referral is itself a compliance posture. A case that should not have been referred and was referred wastes regulator resources; a case that should have been referred and was not produces FCA exposure for the payer.
- **Fix:** Two-part fix:
  1. Derive a deterministic event key from the flag identifier (`flag.flag_id` plus `flag.detector_source`), and use it as a write-once guard in DynamoDB:
     ```
     event_key = flag.flag_id + "|" + flag.detector_source
     try:
         DynamoDB.PutItem(
             table = "processed-flag-events",
             item  = { event_key: event_key, processed_at: NOW() },
             condition = "attribute_not_exists(event_key)"
         )
     except ConditionalCheckFailedException:
         emit_metric("flag_event_duplicate_dropped", 1)
         RETURN
     ```
     The processed-flag-events table can have a TTL of 90 days. This prevents the case-flag append, the subgraph fetch, and the triage-event publish from running twice for the same flag.
  2. Add a "Trigger idempotency" bullet to "Why This Isn't Production-Ready" tying the discipline to the recurring chapter pattern.

  This is now the twelfth consecutive recipe with this finding (2.4-2.10, 3.1-3.6). The cookbook editor should treat the trigger-idempotency appendix as the highest-leverage cookbook-wide editorial investment. The per-recipe editorial loop is producing diminishing returns.

#### Finding A2: No DLQ or Poison-Message Handling for Ingestion, Rules-Engine, Evidence-Aggregator, or Outcome-Capture Lambdas

- **Severity:** MEDIUM
- **Expert:** Architecture
- **Location:** Architecture Diagram (`I[AWS Lambda\nstream-normalizer]`, `T[AWS Lambda\nrules-engine]`, `Y[AWS Lambda\nevidence-aggregator]`, `AG[AWS Lambda\noutcome-capture]` all without `OnFailure` destinations); Prerequisites table.
- **Problem:** No Dead Letter Queue or `OnFailure` destination is configured for any Lambda in the pipeline. The recipe's own "Why This Isn't Production-Ready" section names "Disaster recovery and continuity" as a core production concern with the framing "a quarter without FWA detection is a quarter of undetected schemes." But the per-event poison-message recovery discipline is at a different abstraction level than that paragraph addresses.

  Specific failure modes that surface here:

  - **Stream-normalizer Lambda fails on a malformed message.** The clearinghouse emits an unexpected 837 segment (rare but happens with new payer-specific extensions), the LOINC or CPT code is not in the lookup table (codes change quarterly with CCI updates), the entity-resolution call to NPPES times out. Lambda's default async retry behavior is two retries over six hours and then drop. The result is silently lost: no canonical claim reaches the rules layer, no flags fire on that claim, the claim flows through adjudication without screening. For a system whose value proposition is catching fraud-bearing claims, silently-failing ingestion is the worst possible failure mode.

  - **Rules-engine Lambda fails during rule evaluation** (DynamoDB throttle on the patient-context-cache, S3 read timeout on the rule library, exclusion-check API timeout). Same retry-then-drop pattern. The claim passes through unscored. For a claim that should have triggered a critical rule (post-mortem billing, billing-by-excluded-entity), this is exactly the failure mode the system is meant to catch.

  - **Evidence-aggregator Lambda fails on a malformed flag event** (the supervised re-ranker emits an unexpected schema, the Neptune subgraph fetch fails because of a stale node reference, the OpenSearch put fails because of a mapping issue). The case state is left half-updated, the triage event is not published, the investigator queue is missing a case that should have been there.

  - **Outcome-capture Lambda fails on a malformed outcome event** (the case-management UI emits an unexpected outcome_type value, the confirmed_loss field has a non-numeric value, the supporting_flags reference flag IDs that no longer exist in the flag store). The label is never written, the case stays "open" forever, the supervised re-ranker's training distribution loses the labeled case which is the highest-value training signal.

  Without DLQs, these failures appear only in CloudWatch Logs and require log-trawling to discover. For a system whose flag outputs feed compliance-mandated regulatory referrals, silently-failing scoring is a regulatory-compliance failure mode in addition to an operational one.
- **Fix:** Add four SQS DLQs to the Architecture Diagram: `stream-normalizer-dlq`, `rules-engine-dlq`, `evidence-aggregator-dlq`, and `outcome-capture-dlq`. Add a Prerequisites note: "Configure each Lambda's `OnFailure` destination to a dedicated SQS DLQ. CloudWatch alarms on DLQ depth alert the on-call SIU-engineering team; for the stream-normalizer-dlq specifically, alarm threshold should be 1 because a single dropped claim is a claim that escaped scoring. Replay events from DLQ after fixing the root cause; for events older than the regulatory-referral compliance window, escalate to compliance-team review rather than auto-replay because the timing-of-detection is itself part of the compliance posture under 42 CFR 422.504(h) and 42 CFR 438.608." Add a "DLQ and replay" bullet to "Why This Isn't Production-Ready" that ties the discipline to the existing disaster-recovery paragraph.
