#### Finding S1: SNS Interrupt-Alert Payload PHI Minimization Not Explicit in Pseudocode

- **Severity:** MEDIUM
- **Expert:** Security
- **Location:** Step 5 `route_flags`, the SNS publish at the end (`SNS.Publish(topic = INTERRUPT_ALERT_TOPIC, message = build_pharmacist_alert_payload(anomaly_event), attributes = {...})`); Why-These-Services Amazon SNS paragraph; Prerequisites IAM row.
- **Problem:** The pseudocode's SNS message is constructed by `build_pharmacist_alert_payload(anomaly_event)`, with no explicit specification of what fields are included. The anomaly_event dict (from earlier in Step 5) contains `patient_id`, `drug_rxnorm`, `drug_display_name`, `flags` (which include actual dose values and threshold values), `context_snapshot` (which carries weight, labs, active medications). If `build_pharmacist_alert_payload` serializes the entire anomaly_event into the SNS message, the payload carries a substantial PHI surface through SNS infrastructure, downstream subscribers (the pharmacist's workstation app, the on-call pager channel, mobile push notifications), and any logs those subscribers generate. SMS or pager notifications can be visible on lock screens; Slack or Teams webhook subscribers retain message history; mobile push notifications appear in the device's notification shade.

  Recipe 3.1's expert review settled the chapter discipline: "The message carries the case id only; the analyst UI fetches the full record by id so the notification channel never carries PHI." Recipe 3.3 reaffirmed the convention. The Python companion for 3.4 (per the existing code review's "Healthcare-Specific Requirements" section) does the right thing: "_publish_interrupt_alert builds a message with only event_id, severity, drug display name, and timestamp." But the main recipe's pseudocode doesn't state this discipline, so a reader who copies the walkthrough without reading the Python companion (or a reader who implements in a different language) will not know to constrain the payload.

  The drug display name in the subject line is a minor PHI-adjacency concern that Recipe 3.4's Python companion accepts as a teaching example; for high-stigma medication classes (HIV antiretrovirals, methadone, buprenorphine, gender-affirming hormones, certain psychiatric medications), the drug name itself is a diagnostic disclosure. Lock-screen visibility on a personal device is a real PHI exposure surface, and the patient-safety stakes of medication dispensing don't change that.
- **Fix:** Update the Step 5 pseudocode to make the payload-minimization discipline explicit:
  ```
  // The SNS message carries the event ID, severity, and a coarse routing tier
  // only; the pharmacist UI fetches the full record (drug, dose, patient,
  // context) by ID. PHI does not transit through SNS, downstream notification
  // channels, or any logs they generate. For high-stigma drug classes
  // (antiretrovirals, opioid-use-disorder treatments, gender-affirming hormones,
  // psychiatric medications), even the drug display name is a diagnostic
  // disclosure and should not appear in the notification subject line.
  SNS.Publish(
      topic   = INTERRUPT_ALERT_TOPIC,
      message = {
          event_id:    anomaly_event.event_id,
          severity:    "interrupt",
          routing_tier: anomaly_event.severity,
          fetch_by_id: True
      },
      attributes = {
          "patient_location": enriched_event.patient_location,
          "severity":         "interrupt"
      }
  )
  ```
  Add a one-line note to the Why-These-Services paragraph for SNS naming this convention. Cross-link to the Python companion's `_publish_interrupt_alert` implementation. Same chapter-3-settled minimum-PHI-in-notification pattern as Recipes 3.1 and 3.3.

#### Finding S3: HL7 v2 MLLP Bridge Security Posture Not Specified

- **Severity:** LOW
- **Expert:** Security / Networking
- **Location:** Why-These-Services "Amazon MSK or Amazon MQ for HL7 integration" paragraph ("Most hospital EHR integrations speak HL7 v2 over MLLP (Minimal Lower Layer Protocol), and getting those messages into AWS typically uses an on-premises MLLP receiver that republishes to Amazon MQ (ActiveMQ flavor) or an MSK (Managed Kafka) topic"); Architecture Diagram (`N[On-Prem MLLP Bridge\n+ Amazon MQ]`); Prerequisites EHR Integration row.
- **Problem:** MLLP itself has no native authentication or encryption (it's TCP framing over a stream socket). Production HL7 v2 deployments wrap MLLP in TLS (often called "MLLP over TLS" or "MLLPS") and authenticate connections via mutual TLS with hospital-issued certificates. The on-premises MLLP bridge is the PHI ingress surface from the hospital's EHR into AWS; it traverses the boundary between the hospital network and the AWS VPC via Site-to-Site VPN or AWS Direct Connect, both of which have their own configuration disciplines. The recipe describes the MLLP bridge as "an on-premises MLLP receiver that republishes to Amazon MQ" without addressing:

  1. MLLP-over-TLS vs unencrypted MLLP (the latter is still common in hospital integration engine deployments and is increasingly out of policy).
  2. Site-to-Site VPN vs Direct Connect for the on-premises-to-AWS connection (Direct Connect is the production standard for high-volume PHI ingress; VPN is acceptable for lower volumes).
  3. Mutual TLS or token-based authentication on the MQ broker (the broker is the trust boundary; an unauthenticated broker is a PHI broadcast channel).
  4. Network segmentation: the on-premises bridge should sit in a DMZ or integration tier, not on the hospital's internal clinical network.

  These are standard healthcare-integration disciplines that the recipe assumes the reader has but doesn't name. For a teaching example targeting a mixed audience (executives, architects, engineers, PMs), the omission is more significant than it would be for a pure-engineer audience.
- **Fix:** Add a one-paragraph note to the Why-These-Services Amazon MSK / Amazon MQ paragraph: "The on-premises MLLP receiver is the PHI ingress surface into AWS. Wrap MLLP in TLS (MLLPS) with mutual TLS authentication; deploy the receiver in a DMZ or integration tier, not on the clinical network; connect to AWS via Direct Connect for production volumes (Site-to-Site VPN is acceptable for lower volumes and pilot deployments); authenticate the AWS-side MQ or Kafka broker via mutual TLS or short-lived IAM-derived tokens rather than long-lived shared secrets." Add to Prerequisites EHR Integration row: "MLLP-over-TLS with mutual TLS authentication is the production posture; raw MLLP is acceptable only for development environments with synthetic data."

#### Finding A1: Outcome-Event Idempotency Not Modeled at the Feedback-Capture Lambda

- **Severity:** MEDIUM
- **Expert:** Architecture
- **Location:** Step 7 `on_pharmacist_response` and `on_adverse_event_report` (consumed via EventBridge by the feedback-capture Lambda); Architecture Diagram (`L[Amazon EventBridge\nanomaly-events] --> M3[AWS Lambda\nfeedback-capture]`).
- **Problem:** The architecture is event-driven from the analyst tooling and the incident-reporting system to the feedback-capture Lambda via EventBridge. EventBridge guarantees at-least-once delivery; Lambda asynchronous invocation also retries on failure. If EventBridge redelivers an event (the Lambda's first attempt timed out, the analyst tooling's PutEvents call retried after a transient error, or any of the failure modes documented in the recurring trigger-idempotency findings across Recipes 2.4-2.10, 3.1, 3.2, and 3.3), the feedback-capture runs twice on the same outcome. Each run:

  1. **Updates the OpenSearch anomaly record with the response.** The pseudocode for `on_pharmacist_response` calls `OpenSearch.Update("medication-anomalies", response_event.anomaly_event_id, anomaly)`. OpenSearch updates are idempotent in the document-replacement sense (the second update overwrites the first with the same fields), but a redelivered event with a slightly different `responded_at` timestamp (because the analyst tooling re-emitted the event and used a fresh timestamp) overwrites the resolution timestamp inconsistently, and any downstream consumer of "first-response time" metrics gets noisy data.

  2. **Writes a fresh label row to S3 with a UUID-keyed path.** Both `on_pharmacist_response` (modified-or-cancelled-order labels) and `on_adverse_event_report` (ADE-confirmed labels) call `S3.PutObject(bucket = "medication-anomaly-labels", key = date_partitioned_key(...) + "/" + uuid() + ".parquet", ...)`. The S3 PutObject is "idempotent" only in the sense that two writes don't conflict (different UUIDs), but the resulting two label rows for the same response or ADE will be picked up by the next quarterly retrain. This biases the supervised classifier's training distribution toward whichever cases happened to be retried, and degrades the missed-adverse-event signal which is the recipe's most-emphasized feedback signal.

  3. **Emits duplicate `flag_response`, `missed_adverse_event`, and other CloudWatch metrics**, double-counting the outcome in operational dashboards. For override-rate-driven rule retirement (which the recipe explicitly identifies as part of the rule-tuning loop), doubled override counts directly distort which rules look high-override and get retired. For a patient-safety system this is a higher-stakes consequence than the same finding in 3.1, 3.2, or 3.3: a rule retired because of artificially-doubled override counts is a missed-future-flag, which can be a missed-future-ADE.

  Same recurring trigger-idempotency pattern as Recipes 2.4-2.10, 3.1, 3.2, and 3.3. The fix template is the same: deterministic event-key derivation, conditional-write enforcement.
- **Fix:** Two-part fix:

  1. Derive a deterministic event key from the source event identifier (response_event.anomaly_event_id + response_event.response for pharmacist responses; ade_event.id + dispense.event_id for ADE reports), and use it as a write-once guard in DynamoDB:
     ```
     event_key = response_event.anomaly_event_id + "|" + response_event.response
     try:
         DynamoDB.PutItem(
             table = "processed-feedback-events",
             item = { event_key: event_key, processed_at: NOW() },
             condition = "attribute_not_exists(event_key)"
         )
     except ConditionalCheckFailedException:
         emit_metric("feedback_event_duplicate_dropped", 1)
         RETURN
     ```
     The processed-feedback-events table can have a TTL of 90 days. This prevents the OpenSearch update, the label write, and the metric emissions from running twice for the same event.

  2. Add a "Trigger idempotency" bullet to "Why This Isn't Production-Ready" tying the discipline to the recurring chapter pattern.

  This is now the tenth consecutive recipe with this finding (2.4-2.10, 3.1, 3.2, 3.3, 3.4). The cookbook editor should treat the appendix-consolidation as the highest-leverage cookbook-wide editorial investment. For Recipe 3.4 specifically, the patient-safety stakes elevate the consequence of the doubled-counts to a level that warrants in-recipe discussion even if a cookbook-wide appendix is created.

#### Finding A4: Cross-Reactive Allergy Default Severity Comment Risks Teaching Over-Restriction of Beta-Lactams

- **Severity:** MEDIUM
- **Expert:** Architecture / Domain Accuracy
- **Location:** Step 3 `rule_screen` pseudocode, the `allergy_contraindication` block:
  ```
  "allergy_contraindication":
      FOR each allergen in enriched_event.allergies:
          IF allergen.normalized_id in rule.cross_reactive_allergens:
              flags.append({
                  ...
                  severity:   "interrupt",    // almost always interrupt-severity
                  ...
              })
  ```
- **Problem:** The "almost always interrupt-severity" comment overstates the appropriate severity tier for cross-reactivity-based allergy alerts and risks teaching readers to over-restrict beta-lactam therapy in penicillin-allergic patients. The clinical reality:

  1. **Direct allergy contraindications** (the patient's documented allergen exactly matches the dispensed drug) are almost always interrupt-severity. The patient with documented anaphylaxis to penicillin should not receive penicillin. Hard stop.

  2. **Cross-reactive allergy alerts** are different. The most common case is the penicillin-cephalosporin cross-reactivity alert. Modern evidence has substantially revised the historical 10% cross-reactivity figure: actual cross-reactivity for first-generation cephalosporins is ~1-2%, for third- and fourth-generation cephalosporins is essentially nil, and for carbapenems is <1%. Hard-stop interrupt alerts on penicillin-allergic patients receiving ceftriaxone (a third-generation cephalosporin where cross-reactivity is essentially zero) cause denial of effective antibiotic therapy in 99% of patients who would tolerate it. This is a known patient-safety issue: over-restriction of beta-lactams in penicillin-allergic patients is associated with worse infection outcomes, longer hospital stays, more C. difficile infections, and higher mortality. The Joint Commission and ASHP have published guidance on de-labeling penicillin allergies and avoiding over-restrictive cross-reactivity alerts.

  3. **The right severity tier for cross-reactivity alerts depends on the specific drug pair and the allergic-reaction history.** Penicillin-allergic patients with documented anaphylaxis getting first-generation cephalosporins might warrant interrupt-severity (because anaphylaxis is the high-stakes case). Penicillin-allergic patients with documented "rash-only" or unspecified reactions getting third-generation cephalosporins should generally not be interrupt-severity (the cross-reactivity rate is <1% and the clinical alternative is a fluoroquinolone or vancomycin, both of which have their own significant risks). The "almost always interrupt" framing collapses this clinical nuance into a single severity tier.

  This is a recipe that targets a mixed audience including clinical-informatics teams who may design rule libraries based on the recipe's framing. The "almost always interrupt-severity" comment in the canonical pseudocode risks being copied into rule libraries that then over-restrict beta-lactams.
- **Fix:** Update the pseudocode comment to differentiate direct-allergy vs cross-reactive cases:
  ```
  "allergy_contraindication":
      FOR each allergen in enriched_event.allergies:
          IF allergen.normalized_id == rule.direct_allergen:
              // Direct allergen match (e.g., penicillin-allergic patient
              // receiving penicillin). Interrupt severity is appropriate.
              flags.append({
                  ...
                  severity:   "interrupt",
                  ...
              })
          ELSE IF allergen.normalized_id in rule.cross_reactive_allergens:
              // Cross-reactivity (e.g., penicillin-allergic patient receiving
              // a cephalosporin). Severity depends on (1) the specific drug
              // pair (penicillin/first-gen cephalosporin cross-reactivity is
              // ~1-2%; penicillin/third-gen cephalosporin is essentially nil;
              // penicillin/carbapenem is <1%) and (2) the allergic-reaction
              // history (anaphylaxis vs rash vs unspecified). Defer to the
              // rule's per-pair severity rather than a global "interrupt."
              // See ASHP and Joint Commission guidance on penicillin-allergy
              // de-labeling and beta-lactam stewardship.
              flags.append({
                  ...
                  severity:   rule.cross_reactive_severity[allergen.reaction_type] ?? "synchronous",
                  ...
              })
  ```
  Update the prose in The Technology section's Interaction Anomalies subsection to acknowledge cross-reactivity calibration: "Drug-class cross-reactivity (the classic case is penicillin-cephalosporin, where historical 10% cross-reactivity figures have been revised down to ~1-2% for first-generation cephalosporins and essentially zero for third-generation) requires per-pair severity calibration rather than a single severity tier. Over-restriction of beta-lactams in penicillin-allergic patients is itself a documented patient-safety problem, and the rule library has to encode current evidence rather than historical assumptions."
